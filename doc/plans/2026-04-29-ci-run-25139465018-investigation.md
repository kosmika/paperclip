# CI Investigation: GitHub Actions Run 25139465018

Date: 2026-04-29
Issue: `PAPA-190`
Run: `https://github.com/paperclipai/paperclip/actions/runs/25139465018`
Failing commit: `a4ac6ff133fbe8bdb82f4046fda85f7cb372b6a9`

## Summary

The failing run does not look like an application or test regression from commit `a4ac6ff`.

`verify_canary` passed. `publish_canary` also successfully built and published the packages. The failure happened immediately afterward in the post-publish registry validation step, where the release script expected npm's package document for `@paperclipai/ui` to be fully updated right away.

My working theory is that this is an npm registry propagation race, not a bad publish and not a defect in the sandbox callback bridge changes from `a4ac6ff`.

## Evidence

From the `publish_canary` log:

- The job published `@paperclipai/ui@2026.430.0-canary.0` successfully.
- The script then confirmed `@paperclipai/ui@2026.430.0-canary.0` was visible through `npm view`.
- Seconds later, the registry-state verification failed because the full package document still reported:
  - `dist-tag canary resolves to 2026.429.0-canary.2, expected 2026.430.0-canary.0`
  - `npm registry is missing manifest data for 2026.430.0-canary.0`

That combination matters:

- Version-specific lookup said the new UI package existed.
- Whole-package registry metadata still looked stale.

Those two facts are consistent with eventual consistency or CDN/cache lag on the npm registry document, especially for a large package with many historical versions like `@paperclipai/ui`.

## Why this points at release verification, not the product change

1. `verify_canary` passed, so the repo still built and tested cleanly.
2. The failure happened after successful publication, during registry verification.
3. The log names only npm metadata mismatches for `@paperclipai/ui`.
4. The failing commit changed sandbox callback plumbing, not release automation.

## Probable failure mechanism

At commit `a4ac6ff`, `scripts/release.sh` runs two separate post-publish checks:

1. Wait for `npm view <pkg>@<version>` to resolve.
2. Run `scripts/verify-release-registry-state.mjs`, which fetches the package document from `https://registry.npmjs.org/<package>` and immediately asserts:
   - the release dist-tag points at the new version
   - `packageDoc.versions[targetVersion]` exists
   - internal `@paperclipai/*` dependencies are visible in the fetched package docs

The problem is that step 2 does not retry. It assumes the package root document, its `dist-tags`, and its `versions` map are fresh as soon as version-specific lookups are fresh. The log shows that assumption was false for `@paperclipai/ui`.

## Most likely root cause

The release workflow is too strict about immediate npm registry consistency after publishing. It treats transient propagation lag as a hard failure.

## Recommended fix

Harden the post-publish verification instead of changing application code.

Recommended direction:

1. Add retry/backoff around `verify-release-registry-state.mjs` instead of running it once.
2. Consider separating checks:
   - version-specific availability
   - dist-tag convergence
   - full package-document dependency verification
3. Prefer version-specific manifest fetches where possible instead of relying only on the package root document's `versions` map.

## Expected outcome after fix

Future canary publishes should wait out short npm propagation delays instead of failing after a successful publish.
